## Critical conventions (do not violate)

These are non-negotiable architectural commitments from the specifications:

- **LPM byte-preservation**: Source.content is byte-for-byte verbatim from input. No normalisation, no encoding transformation, no whitespace collapse beyond canonical character-stream conversion. Implemented via Pydantic `frozen=True` on the Source model — attempted modification raises RuntimeError. This is the central LPM enforcement; do not weaken it.

- **Mode declaration via decorator**: every mechanism Pass uses `@pass_mode("LPM")` (or appropriate mode). The decorator records `mode_active` on AnalysisPass, instruments execution to detect mode violations, and sets `execution_status=Failed` on violation. Implement the decorator in `core/modes/` per Row 4 Applied §9.

- **Mechanism provenance lives on AnalysisPass.outputs**: per Issue F4 architectural commitment, mechanism-internal data (Read Witness, mechanism_data, mode_violations) lives on `AnalysisPass.outputs` (JSONB column), NOT on canonical entity types themselves. See Row 4 Applied §10 and Row 4 Mechanism Source Capture §7.

- **One transaction per mechanism execution**: per Row 4 Applied §5 — all entities (Sources, Segments, SourceAtoms, AnalysisPass) commit atomically per mechanism execution, or none do. Exception: on failure, the AnalysisPass record with failure detail is committed in a separate transaction so failures remain auditable.

- **Identifier sequencing**: per Row 4 Applied §5 — Postgres sequences per identifier prefix (src_id_seq → SRC###; seg_id_seq → SEG###; etc.). Zero-pad to 3 digits; format extends automatically as numbers grow. The exact prefix for atom IDs is pending finalisation per Issue F11; recommended ATM###.
