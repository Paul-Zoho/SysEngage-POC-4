# SysEngage

AI-powered Systems Engineering Tool implementing Zachman framework analysis. Helps Practitioners produce structured systems-engineering analysis (Rows 1-6, multi-perspective) of software products through deterministic and AI-assisted mechanisms operating on supplied source material.

## What this is

SysEngage analyses a product (the "described system") across the Zachman framework's six rows of perspective, producing a canonical ledger of entities (Sources, Signals, ZachmanCells, CellContentItems, Domains, Requirements, Stakeholders, etc.) representing the analysis. The Practitioner supplies input material; SysEngage processes it through a sequence of Phases and mechanisms; outputs are canonical ledger entries with full audit trail.

This v1 prototype implements the foundational mechanism (Source Capture) end-to-end. Subsequent mechanism implementations will follow the same architectural pattern.

## Tech stack

- **Language**: Python 3.12+
- **Web framework**: FastAPI
- **Persistence**: Neon PostgreSQL via SQLAlchemy 2.x
- **Schema migrations**: Alembic

## Critical conventions

These are non-negotiable — they encode architectural commitments from the Row 4 specifications.

### LPM byte-preservation discipline

Phase 1 Source Capture operates strictly under LPM (Literal Persistence Mode). `Source.content` is byte-for-byte verbatim from input — **no normalisation, no encoding transformation, no whitespace collapse**. Once captured, `Source.content` is immutable in memory (Pydantic `frozen=True`). Attempted modification raises `RuntimeError`.

### Mode declaration via decorator

Every mechanism Pass declares its Transformation Mode via decorator:

```
@pass_mode("LPM")
def pass_0_read_witness(...):
    ...
```

The decorator records `mode_active` on `AnalysisPass`, instruments execution to detect mode violations, and sets `execution_status=Failed` if violation detected. Modes: `LPM`, `IM`, `DM`, `CM` (combinations possible — see `core/modes/`).

### Ledger transactional discipline

Each mechanism execution is wrapped in **one Postgres transaction**. All produced entities (Sources, Segments, SourceAtoms, AnalysisPass) commit atomically — or none do. On failure, transaction rolls back; only the `AnalysisPass` record with failure detail is committed (in a separate transaction) so failures remain auditable.

### Mechanism provenance lives on AnalysisPass

Per architectural decision F4: mechanism-internal data (pattern matches, intermediate data, debug info) lives on `AnalysisPass.outputs` (a JSONB column), **not on the canonical entity types themselves**. This preserves canonical schema purity while keeping audit trail rich. Sub-structures:

- `AnalysisPass.outputs.read_witness` — Phase 1 Read Witness data (per F10)
- `AnalysisPass.outputs.mechanism_data` — mechanism-specific data per Implementation Spec
- `AnalysisPass.outputs.mode_violations` — array of violations detected (empty on compliant execution)

### Identifier sequencing

Per canonical ledger spec v2.9, entity identifiers follow per-prefix conventions. Implementation uses Postgres sequences:

- `src_id_seq` → `SRC###` (Source)
- `seg_id_seq` → `SEG###` (Segment)
- `atm_id_seq` → atom identifier (format pending finalisation per F11; recommended `ATM###`)
- `pass_id_seq` → `AP###` or `PASS###` (AnalysisPass — recommendation pending)

Zero-pad to 3 digits; format extends automatically as numbers grow (`SRC1000`, `SRC10000`, etc.).

### Schema validation discipline

Every entity validates twice:

1. **On creation** — Pydantic constructor validates against canonical schema rules
2. **On persistence** — SQLAlchemy ORM events validate before insert

Cross-entity referential integrity enforced via Postgres foreign keys (database-level) and Pydantic validators (application-level invariants).

## Source Capture v1 — what to build first

The first deliverable is the Source Capture mechanism end-to-end:

1. Schema models (Pydantic + SQLAlchemy + Alembic migrations) for `Source`, `Segment`, `SourceAtom`, `AnalysisPass`
2. Mode discipline decorator (`core/modes/`)
3. Audit trail helpers (`core/audit_trail.py`) for AnalysisPass record management
4. Source Capture mechanism (`mechanisms/source_capture/` — one Python module per Pass)
5. FastAPI route for input upload + minimal post-execution summary view (`api/source_capture.py` + `templates/source_capture/`)
6. pytest tests for each Pass + end-to-end mechanism (`tests/source_capture/`) using fixtures in `fixtures/source_capture/`
7. Sequential orchestrator (`core/orchestrator.py`) — invokes Source Capture for the v1 single-mechanism scope

Verification target: run `pytest tests/source_capture/` and pass all fixtures (happy path + edge cases + determinism + Phase 10 re-execution per Implementation Spec §9).

## What NOT to build for v1

These are explicit non-goals for v1:

- Authentication / multi-tenancy (single-tenant prototype)
- JS framework / SPA frontend (Jinja templates suffice for Source Capture UI)
- Other mechanisms beyond Source Capture (one mechanism end-to-end first)
- AI Model integration (Source Capture has no AI involvement; downstream mechanisms add it)
- Phase 10 re-execution UI (re-execution logic supported per Implementation Spec §10.4 but no UI for v1)
- Caching / cost-control infrastructure (deferred until AI mechanisms are added)
- Workflow engine / event-driven coordination (sequential runner suffices for v1)
- GitHub-as-Practitioner-input-source (file upload only for v1; GitHub repo scanning is candidate v2+ feature per F12 Q1)

## Working with this codebase

A few patterns to follow:

- **Use the deterministic style**. Source Capture is fully deterministic — pure functional Pass logic, isolated side effects, predictable outputs. Tests verify bit-identity across re-runs.
- **Test fixtures are authoritative**. If implementation behaviour differs from fixture expectations, either the implementation is wrong or the fixture needs updating — but the disagreement requires explicit resolution.
- **Verbatim is verbatim**. Source content modification — even "just normalising whitespace" — violates LPM. The Pydantic `frozen=True` enforcement will catch this at runtime, but the discipline matters in code review too.
- **Reference docs over reimplementation**. If something feels under-specified, check Row 4 Applied or the Mechanism Implementation Spec first; they likely cover it. Surface genuine gaps as Issues Tracker findings rather than inventing answers.
